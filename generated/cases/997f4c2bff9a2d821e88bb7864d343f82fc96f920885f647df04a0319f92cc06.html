Scalable Cloud-Native Storage
Systems
1. Abstract
Cloud-native storage systems are essential tools in response to the impressive growth of
big data in the digital universe. In this paper, we investigate the architecture, scalability,
fault tolerance, and cost of popular cloud-native storage systems using AWS S3, Azure
Data Lake Storage Gen2, and Google Cloud Storage.
The investigation utilized an analytical approach. This study revealed 8 scholarly articles
(from 2017 - 2025) and conducted real-world performance assessments of the systems.
The primary findings include: The systems constructed around an "object" storage
paradigm and distributed metadata management feature scalable; regarding fault
tolerance, the combination of erasure coding and data replication at multiple sites yield
lower failure.
2. Introduction
2.1 Background
https://rivery.io/blog/big-data-statistics-how-much-data-is-there-in-the-world/
By 2025, global data is expected to reach 181 zettabytes as more IoT devices, real-time
analytics applications, and AI applications become available.
Traditional storage systems that sit in one particular location (e.g., on-premises servers)
have trouble keeping up with the resulting data growth. Traditional storage systems have
two main issues: (1) there is an upfront cost to purchase and implement a storage system,
and (2) changing the amount of storage used, if needed, is not easy.
Cloud-native storage systems are unique in that they are designed specifically for cloud
deployments that can span multiple locations. They address concerns about traditional
storage systems through the use of three things: object storage, microservices, and a
consumption-based pay-as-you-use model.
2.2 Relevance
For companies in every possible industry, cloud-native storage has become an
imperative and no longer just an option.
Hospitals and clinics need to both store and retrieve massive amounts of medical image
data, often petabytes of it. Banks and financial institutions have rules that require their
storage to be immutable, or not-changeable. And transportation companies depend on
fast (low-latency) storage for real-time GPS and sensor data.
If these companies are going to make sensible technology decisions, they need to
understand the benefits and drawbacks of each of the most popular cloud-native
storage options.
2.3 Research Objectives
 Examine the internal design of AWS S3, Azure Data Lake Storage Gen2, and Google
Cloud Storage.
 Evaluate their capacity for scaling, their resiliency to failure, and their
cost-effectiveness - in published research and industrial performance testing.
 Discuss their use in healthcare, finance, and transport.
 Recommend how best to engineer cloud-native storage for deployment.
3. Literature Review
3.1 Architectural comparison
The core component of a cloud-native storage system is object storage, which stores
data as objects (with associated metadata/identifier). It efficiently scales because it does
not enforce a strict hierarchy so a distributed metadata service can help avoid hot spots.
Object storage has separate control and data planes, both of which can be implemented
as REST/HTTP microservices [1][7].
AWS S3 provides buckets/keys stores that can scale the metadata layer to avoid hot
spots and has storage classes for data hierarchy. Access control is managed via IAM,
which is tightly integrated with the analytics functionality of other AWS services [2].
Azure ADLS Gen2 is built on Azure Blob Storage, but it allows a formal hierarchical
namespace (HNS) for files and folders to organize data. It has POSIX-like ACL
capabilities, tightly integrated with Azure AD for enterprise authentication, and
redundancy [2][4][6], making it more suitable for analytics.
Google Cloud Storage (GCS) uses buckets/objects but as part of a global/distributed
infrastructure. Security and access control is managed via Cloud IAM and APIs are
simple to develop against. GCS is similar to S3 but the defaults and tools differ [2].
The flat object storage namespaces (S3, GCS) can scale easily, while ADLS Gen2’s HNS
supports analytics more easily, but requires additional metadata beyond the base object.
Management of metadata will be essential to scaling beyond a certain threshold.
There is latency incurred with the operation of object storage , but this latency trade-off is
necessary to benefit from durability and availability [5][7].

We employ a straightforward comparative approach. First, we review 8 papers from 2017
to 2025 that discuss cloud-native storage, architecture, scalability, fault tolerance, and

Research paper inclusion: we include peer reviewed articles, conference papers, and a

Data extraction: For every system, we collect only simple fields: storage model (object or

5. Findings and Discussion
5.1 Architecture Comparison
We have reviewed AWS S3, Azure Data Lake Storage Gen2, and Google Cloud Storage
in terms of their underlying design. The three services provide a comparable object store
design with separation of the control plane (API, auth, metadata) from the data plane
(object bytes). The metadata and access control depend on distributed services, while
the storage nodes perform durability and scale - similar to earlier systems like DuraCloud
[1 ][2][7].
AWS S3 has a flat namespace that contains buckets and keys. In this service, metadata
is partitioned from operations to prevent hot spots and storage classes allow for flexibility
around the retrieval of data. IAM is the access control for this service which highlights
that the operations of the API itself are purely object operations. The overall systems
design promotes durability and horizontal scale while attempting to maintain a simple
path semantic [2][7].
Azure Data Lake Storage Gen2 extends Blob storage, however, it includes a hierarchical
namespace (HNS). The paths look like folders and files and the ACLs function just like
POSIX. Authentication utilizes Azure Active Directory that like the built in HNS access
enhancement. Azure itself provides zone and geo redundancy to protect the availability
of data during a failure which supports analytics and enterprise policy quite well [2][4][6].
In Google Cloud Storage, objects reside in buckets, in a flat object model, and utilize
shared global infrastructure. Google Cloud Storage has IAM for access restrictions and a
simple REST API (JSON/XML). Google Cloud Storage also works with third-party
vendors that support Google Cloud Storage and integrates well with Google analytics
tools. In comparison, Google Cloud Storage and AWS S3 are generally parallel in the
fundamental premises, but have different defaults and provide different ecosystems [2].
High-Level Overview: hierarchal namespace storage can support workloads that scan
large directories or wish to introduce fine-grained ACLs; however, it also adds more
overhead work in metadata. Flat namespaces can horizontally scale with ease and
support a more straightforward object workflows. Distributed metadata and microservice
control planes can help remediate the choke points that are commonly known with
legacy file stores [1][2][6][7].
Consistency and latency: object requests will traverse multiple front ends, and multiple
metadata / routing services before reaching the object itself. Each hop, or a cross-region
route will add latency in front. However, this same design will also increase durability and
availability with large objects [5][7].
