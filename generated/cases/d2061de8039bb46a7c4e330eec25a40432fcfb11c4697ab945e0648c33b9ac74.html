Frequently Asked Questions
How should I interpret Turnitin's AI writing percentage and false positives?
The percentage shown in the AI writing report is the amount of qualifying text within the submission that Turnitin’s AI writing
detection model determines was either likely AI-generated text from a large-language model or likely AI-generated text that was
likely revised using an AI paraphrase tool or word spinner.
False positives (incorrectly flagging human-written text as AI-generated) are a possibility in AI models.
AI detection scores under 20%, which we do not surface in new reports, have a higher likelihood of false positives. To reduce the
likelihood of misinterpretation, no score or highlights are attributed and are indicated with an asterisk in the report (*%).
The AI writing percentage should not be the sole basis to determine whether misconduct has occurred. The reviewer/instructor
should use the percentage as a means to start a formative conversation with their student and/or use it to examine the submitted
assignment in accordance with their school's policies.
What does 'qualifying text' mean?
Our model only processes qualifying text in the form of long-form writing. Long-form writing means individual sentences contained in paragraphs that make up a
longer piece of written work, such as an essay, a dissertation, or an article, etc. Qualifying text that has been determined to be likely AI-generated will be
highlighted in cyan in the submission, and likely AI-generated and then likely AI-paraphrased will be highlighted purple.
Non-qualifying text, such as bullet points, annotated bibliographies, etc., will not be processed and can create disparity between the submission highlights and the
percentage shown.

Part I: Extensions to Task Requirements
To improve the utility of the application for analysis and to improve the user
experience, additional functionality has been added that could be classified as
enhancements of the original functional requirements of the task. The goal of this
additional functionality is to provide users with more options for how to analyze their
results and to provide users with easier ways to access the data present in the
application’s dataset.
 Multiple Hash Functions: The GUI contains a dropdown box for selecting which
hash function to use. The application supports this capability in its underlying
framework, and the current implementation only provides the polynomial rolling
hash function; therefore, it is possible to evaluate the relative quality of the
polynomial rolling hash function’s ability to distribute n-grams.
 Vocabulary Sorting Options: Users may sort their vocabulary lists either
alphabetically by word or numerically by the frequency of the word. This addition
greatly enhances the user's ability to review the displayed information and
analyze how their vocabulary list is distributed and what specific words have
been used.
 CSV Export Functionality: This allows users to save the entire vocabulary list
(word and frequency) as a csv file. This functionality makes it easier to store and
analyze data that can be imported into spreadsheet or other analysis-based
tools.

Part II: System Design and Implementation
This report continues with a description of the software architecture design, core
data structures, and primary algorithms used to implement a system that meets the
requirements of this particular project. The design of all aspects of the N-Grams
system was primarily focused on developing a system that has strong Encapsulation,
Modularisation and Computing Efficiency in processing text and generating the
n-gram probabilistic language models.
Core Data Structures
The Language model utilizes a customised hash table to effectively store and retrieve
n-gram Freqencies. There are two main classes which comprise the Hash Table:
MyHashTable: It implements implements a hash table based on separate chaining
for collision resolution with its main components being an array of MyLinkedObject
instances that are referred to as "buckets" and a hash function to determine which
bucket holds a specific word sequence. Whenever you add a word sequence, the
hash function will return an index value to identify the proper bucket; the word
sequence is then linked to that bucket by way of a linked list being appended at that
index. The MyHashTable public interface hides all the underlying list and node
structures so users can simply call either the add(String) method to store a word
sequence or use the getCount(String) method to retrieve the count for a word
sequence from the appropriate bucket.
MyLinkedObject: It represents each of the linked-list nodes that compose the hash
table for each bucket. Each instance of MyLinkedObject has an instance variable to
keep track of the word (word) that corresponds to the instance, the count of
occurrences (count) for the word, and a link to the subsequent node in the chain
(next). A significant design decision in designing MyLinkedObjects was to create the
structure in such a way that the method setWord adds a new instance of
MyLinkedObject into the list, maintaining the word order as specified on the
assignment's design considerations. In order to accomplish this, MyLinkedObject
makes adding a new instance a pre-sorted operation. This optimization has proven
to be beneficial, as once the user clicks on the button to sort the complete set of
words (the vocabulary) in alphabetical order, the application does not have to sort
the complete vocabulary; it simply walks through the already-sorted linked list and
presents the results to the user almost instantaneously.
Key Algorithms and Logic

The Hashing and Tokenization Processes begin with the loading of the input file and
Tokenizing of it. As required by the assignment, Tokenization Logic converts all
punctuation to whitespace. Many tokens are present in the document of various
Capitalizations or Spellings, e.g., Have, Has; these tokens will have separate entries
in the Tokens List. Once the tokens have been Tokenized and stored in the table of
MyHashTable, each Token is passed to the PolynomialRollingHash function, which
determines the hash value and maps it into a bucket in the array of buckets of the
MyHashTable
When building a Language Model, the application creates three Language Models:
Unigram, Bigram, and Trigram. In order to do this, the application loops through the
Tokenized Document and gathers sequences of one (or two and three) words. Each
sequence will have its own frequency tracked in the corresponding Language Model’
s Hash Table. To build the Bigram Model, for example, every word wk is looped
through and paired with its previous word wk-1, and then the count of that pair (wk-1,
wk) is incremented in the Bigram Hash Table. The resulting Hash Tables will contain
the raw count of token pairs (for example, c(wk-1), for Unigrams, and c(wk-1,wk), for
Bigrams) required to obtain the probability.

Part III: Discussion
Task 1: MyLinkedObject methods
The constructor MyLinkedObject(String w) creates a node with a word set to W, a
count of 1, and no next node. The method setWord(String w) is designed to keep the
linked list in alphabetical order as new words are added to it. If w matches a node's
word, it increases that node's count by 1; if the next node is null, it adds a new node
into the linked list after the current node; if w is alphabetically before the word of the
next node, it inserts the word at that location, otherwise, it recursively follows the
linked list to the next node. If the word being added is alphabetically before the
current node's word, it throws an exception. The method isWord(String w) checks if
the word of the current node matches w, and returns true if it does, otherwise it
checks the next node in the linked list recursively. The method getCount(String w)
searches each node until it finds a match for w and returns that node's count, or
continues the search recursively, returning 0 if no match is found..

Task 2: MyHashFunction and chosen algorithms
An interface for a hash function is defined by the MyHashFunction class. In the
interface, there are two methods, one is int hash(String seq, int m) and the other is
normalize(String seq). The hash for the first letter is calculated by modulus m of the
Unicode corresponding to the first character in the string. The hash of the string, via
polynomial rolling hash method, is calculated by way of using a base of 31 via the
expression h = (h * base + char) % m. The DJB2 variant of the polynomial rolling hash
applies the iterative method via the expression of h = h * 33 + char. In general, when
using a larger value for m, then the average number of tokens would be lower than if
the integer value for m was smaller. In addition, better types of hashing functions
distribute items in the buckets more evenly than that of the hash of the first letter of
the string.
