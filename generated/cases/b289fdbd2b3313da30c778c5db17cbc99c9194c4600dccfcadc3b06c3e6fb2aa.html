1. Storage Infrastructure Management (Multiple Choice/Short Answer)
1.1 Core Purpose & Key Activities
Purpose: Ensure business continuity; proactive meeting of service levels via management processes
and tools.
Key Management Activities: 5 categories (must distinguish for MCQs)
. Availability Management: Establish config guidelines based on service levels (critical
business needs highest availability).
Example components for critical servers: 2+ HBAs, multipathing software, server
clustering, 2 independent fibre channel switches, RAID protection, backup, local/remote
replication.
. Capacity Management: Ensure adequate resources per service level; includes 2 core
analyses:
Capacity analysis: Regularly compare allocated vs. forecasted storage.
Trend analysis: Track actual storage utilization and consumption rate.
Examples: Storage provisioning (LUN masking on arrays, SAN zoning), estimating future
resource needs (data gathering/analysis).
. Performance Management: Ensure optimal component efficiency; validate performance for
new apps/servers.
Server checks: Volume config, DB design, app layout on HBAs, multipathing software.
SAN checks: Sufficient ISLs (Inter-Switch Links) with adequate bandwidth.
Storage array checks: RAID type, LUN layout, front-end/back-end ports, LUN masking.
. Security Management: Prevent unauthorized access/config.
Examples: User account/access policy management, SAN zoning, LUN masking.
. Reporting: Track/gather component/process info for 4 key goals:
Trend analysis/capacity planning (storage/file system/DB utilization).
Configuration/asset management (device allocation, replicas, equipment details: value,
purchase date, lease status).
Chargeback (storage allocation/utilization by dept/user group).
Performance (component performance metrics).

2. Storage Devices (High-Frequency MCQs: Characteristics & Performance)
2.1 File Systems (Core Abstractions for MCQs/Short Answer)
Definition: Abstraction on persistent storage (magnetic disk/flash); hides physical storage
limitations.
Key Functions (Must Memorize):
. Persistence: Data survives machine crashes, disk block corruption, flash wear.
. Naming: Uses named data (vs. disk block numbers), directories (vs. flat storage), byte-
addressable data (vs. device block-orientation).
. Performance: Cached data, optimized data placement/data structures.
. Shared Access Control: Controlled access to shared data.
File System Abstraction Details:
File: Named linear byte sequence (read/write or memory-mapped).
Directory: Hierarchical group of named files/subdirectories (maps names to file metadata).
Path: Unique file/directory identifier (e.g., /cse/www/education/courses/cds502/2au).
Links:
Hard link: Links name to metadata location.
Soft link: Links name to an alternate name.
Mount: Maps a name in one file system to the root of another.
2.2 Magnetic Disks (MCQs: Structure, Performance, Scheduling)
2.2.1 Core Characteristics (Distinguish from Flash)
Pros: Large capacity, low cost, block-level random access, rare corruption.
Cons: Slow random access, better for streaming access.
2.2.2 Physical Structure (Visualize for MCQs)
Disk surface: Circular, magnetic-coated.
Tracks: Concentric rings (1 micron wide; 100K tracks on 2.5вҖқ disk); separated by guard regions
(reduce corruption).
Outer tracks: More sectors/track вҶ’ higher bandwidth; only outer half of radius used.
Sectors: Arc of a track (min transfer unit); include error-correcting codes (ECC), support 2 sparing
methods:
Sector sparing: Remap bad sectors to spares (transparent).
Slip sparing: Remap all sectors (preserve sequential behavior).
Cylinder: Same track across all platters in a disk pack.
Disk arm: Carries read-write heads; read/write needs 3 steps: seek cylinder вҶ’ wait for sector
rotation вҶ’ transfer data.

2.2.3 Disk Controller (Key Components)
External connections: IDE/ATA, SATA, SCSI (Ultra-160/320), Fibre Channel (FC).
Cache: DRAM to buffer data (e.g., Hitachi disk has 16MB); some RAM stores firmware (embedded
OS).
Functions: Read/write details, cache replacement (usually LRU), failure detection/recovery.
2.2.4 Disk Performance (Formula for Short Answer/MCQ Calculation)
Disk Latency = Seek Time + Rotation Time + Transfer Time
. Seek Time: 1-20ms (time to move arm to track; includes head вҖңsettlingвҖқ).
. Rotation Time: 4-15ms (depends on disk price); average = ВҪ rotation time.
. Transfer Time: 5-10 Вөsec/sector (disk head rate: 50-100MB/s); host rate depends on I/O
connector (USB/SATA).

2.3.1 Core Characteristics (Compare to Magnetic Disk)
Pros: Rare corruption, good read performance, block-level random access.
Cons: Intermediate cost (50x disk), worse random write performance (needs pre-erasure).
2.3.2 Key Limitations & Solutions
Write Limitation: Cannot update in place; must erase blocks first (erasure block: 128-512 KB,
erasure time: several ms); writes/reads in pages (2-4 KB, 50-100 Вөsec).
Solutions:
. Flash Translation Layer (FTL): Firmware maps logical page # to physical location; handles:
Garbage collection: Copy live pages to new block, erase old block (efficient if same-file
blocks are grouped).
Wear-leveling: Limit writes per physical page (prevents early failure).
Sector sparing: Remap faulty pages.
. TRIM Command: File system tells device which blocks are no longer in use (enables FTL to
optimize erasure).
