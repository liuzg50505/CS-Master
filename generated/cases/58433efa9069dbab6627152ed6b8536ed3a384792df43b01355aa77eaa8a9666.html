Formally…why functions?
Abstraction and Reusability
▪ Suppose you write some code that does something useful.
▪ Suppose that code is one you need often, in many different locations
within your application.
▪ Functions represent an easy solution, just call the function without a
need to repeat code.
▪ The abstraction of functionality is an example of theDon’t Repeat
Yourself (DRY) Principleof software development. This is arguably the
strongest motivation for using functions.
Formally…why functions?
Modularity/ decomposition
▪ Functions allow complex processes to be broken up into smaller
steps.
▪ Divide code into modules
– are self-contained
– used to break up code
– intended to be reusable
– keep code organised
– keep code coherent

Function components
▪ Every function has:
– Name
– Parameters (0, one or many)
– Body (what do you want to do)
– A value to return (or void)
– Description or docstring (optional, but highly recommended for good coding)
▪ The function is implemented when it is called only.
