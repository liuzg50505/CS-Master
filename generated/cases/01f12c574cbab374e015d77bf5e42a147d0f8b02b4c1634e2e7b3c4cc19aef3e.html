Implementation of the Dependently Typed
Functional Languageerictt
Eric Finster
February 1, 2026
Abstract
This report details the design and implementation of a small dependently typed functional
language callederictt. We explore the convergence of programming and logic, describe the im-
plementation and main features of the language, describe the typing-rules and provide examples
of the language in action.
1 Introduction
1.1 Motivation and Problem Statement
1.1.1 The Reliability Gap
In traditional software development, there is a persistent gap between the specification (what the
code is supposed to do) and the implementation (what the code actually does). Standard type
systems—while helpful for catching "primitive" errors like adding a string to an integer—are often
too weak to express the logic of the problem domain. This forces developers to rely on extensive
unit testing and runtime assertions to ensure correctness.
1.1.2 The Solution: Dependent Types
Thegoalofthisprojectistoimplementalanguagewherethetypesystemispowerfulenoughtoserve
as the specification itself. By allowing types to depend on values, we move the verification process
from runtime (where errors cause crashes) to compile-time (where errors prevent the program from
even being built).
1.1.3 Problem Statement
Despite their power, dependently typed languages often suffer from high entry barriers due to
extreme syntactic complexity. This implementation seeks to provide a minimal, extensible core for
a dependently typed functional language, focusing on:
•Implementing a bidirectional typechecking algorithm
•Providing implicit arguments and pattern unification
•Using the technique of Normalization by Evaluation (NbE) for term reduction.
1
1.2 State of the Art: Proof Assistants
The theoretical foundations implemented inericttare shared by several industrial-strength proof
assistants. These tools, though varying in their surface syntax and specialized features, all rely on
a dependently typed core to manage the relationship between programs and proofs.
•Coq:One of the oldest and most stable proof assistants, based on the Calculus of Inductive
Constructions. It is widely known for its powerful tactic language,Ltac, which allows for
sophisticated proof automation.
•Agda:A dependently typed functional programming language that emphasizes "program-
ming as proving."ericttshares several architectural similarities with Agda, including the
use of holes (_) and implicit arguments.
•Lean:A newer interactive theorem prover developed at Microsoft Research. It has gained
massive popularity in the mathematical community due to its performance and the "Mathlib"
project, which aims to formalize a vast library of mathematical knowledge.
1.3 Milestones in Formalization
The justification for the implementation complexity of dependently typed systems is found in their
historic successes. These milestones prove that formal verification can tackle problems that are
beyond the reach of traditional peer review or unit testing.
1.3.1 Formalized Mathematics
•The Four-Color Theorem:In 2005, Georges Gonthier used Coq to provide a fully mech-
anized proof of this theorem, which states that any map can be colored with just four colors
such that no two adjacent regions share a color.
•The Odd Order Theorem (Feit–Thompson):Also completed by Gonthier and his team
in 2012, this involved formalizing one of the most complex proofs in group theory, spanning
hundreds of pages of traditional mathematical text.
•TheKeplerConjecture:FormalizedintheFlyspeck Project(usingHOLLightandIsabelle),
this confirmed the densest possible way to stack spheres.
1.3.2 Verified Software Engineering
•CompCert:A high-performance, formal-verified compiler for the C programming language.
It is written and proved in Coq, ensuring that the generated executable code exactly matches
the semantics of the source code, effectively eliminating compiler-introduced bugs.
•seL4 Kernel:The world’s first operating system kernel with a machine-checked proof of
implementation correctness. This ensures the kernel is immune to classes of vulnerabilities
like buffer overflows and pointer errors.

2 Theoretical Background
2.1 The Curry-Howard Isomorphism
The foundational principle underlyingericttis theCurry-Howard Isomorphism, also known as the
"Propositions-as-Types" correspondence. This observation establishes a direct structural mapping
between formal logic and computational type theory.
Under this isomorphism, atypeis viewed as alogical proposition, and aterm(or program)
is viewed as aformal proofof that proposition. If we can construct a termtsuch thatt:A, we
have effectively proven that the propositionAis true.

2.1.2 Computation as Proof Simplification
A unique consequence of this isomorphism is thatnormalization(the process of reducing a term
to its simplest form) corresponds exactly toproof normalization(eliminating detours in logic).
Inerictt, the use of Normalization by Evaluation (NbE) ensures that we can determine if two
proofs (terms) are equivalent. If two programs reduce to the same normal form, they are considered
proof-theoretically identical. This is why the conversion check inunify.mlis the engine of the
entire proof system: it determines if two seemingly different propositions are, in fact, the same.
2.1.3 Dependent Types and Predicate Logic
While "simple" type systems (like those in Haskell or ML) can represent propositional logic, they
lack the power to representpredicate logic. Dependent types bridge this gap.
Inerictt, aΠ-type (PiT) allows the type of a function’s output to depend on thevalueof its
input. Logically, this is equivalent toUniversal Quantification(∀). For example:
•Logic:For every natural numbern, there exists a proof thatn+ 0 =n.
•erictt Type:Pi (n : Nat). Equal (add n zero) n
By checking that a program inhabits this type,ericttverifies the validity of the underlying math-
ematical statement.

4 Implementation oferictt
The implementation ofericttis structured as a modular OCaml project, focusing on a clean
separation between the surface language, the internal core syntax, and the semantic domain used
for evaluation and unification.
4
4.1 Internal Syntax and Representation
The core oferictt(defined interm.ml) uses a unified representation for terms and types, following
the tradition of the Calculus of Constructions. To handle variable binding efficiently, the language
employs a dual-system for variable tracking:
•DeBruijnIndices:Usedintheinternalsyntax(term)torepresentvariablesbytheirdistance
from their binder. This simplifies alpha-equivalence checks.
•De Bruijn Levels:Used in the semantic domain (value) to represent variables as absolute
offsetsfromthebottomofthestack. Thispreventstheneedforexpensive"shifting"operations
during evaluation.

4.3 Bidirectional Type Checking
A central challenge in dependently typed languages is that type synthesis (determining a type from
a term) is often undecidable or leads to ambiguous results, especially in the presence of implicit
arguments and higher-order unification. To address this,ericttimplements abidirectional type-
checkingalgorithm.
4.3.1 The Two Modes: Checking and Inference
Instead of a single typing judgmentΓ⊢t:A, the bidirectional approach splits the process into two
distinct modes:
1.Inference Mode (Γ⊢t⇒A):The system takes a termtand produces a typeA. This
is used for "introductory" components where the type is self-evident or determined by the
context.
2.Checking Mode (Γ⊢t⇐A):The system takes a termtand a known typeA, and verifies
that the term satisfies that type. This is more powerful than inference because the typeA
can "guide" the elaboration of the termt.
4.3.2 Canonical vs. Elimination Forms
The efficiency of this approach inericttstems from the observation that different "shapes" of
terms naturally lend themselves to different modes:
5
Checking Canonical Forms:Canonical forms (Introduction forms), such asλ-abstractions
(LamT) or pairs, do not have a unique type in isolation. For instance,λx.xcould be Nat→Nat
or Bool→Bool. Inerictt, these are alwayscheckedagainst an expected type. This allows the
system to extract the domain type directly from the expectedΠ-type, rather than forcing the user
to annotate the binder.
Inferred Elimination Forms:Elimination forms, such as function applications (AppT) or vari-
ables (VarT), typically have types that can be determined from their components. For an application
f a, weinferthe type off. Once we knowf: Π(x:A).B, we can then switch tocheckingmode
to ensure the argumentamatches the expected domainA.
4.3.3 The Benefits of Bidirectionality
•Reduced Annotation:By propagating type information downward, the user rarely needs
to write types for local variables or intermediate terms.
•Implicit Argument Resolution:Bidirectionality is essential forerictt’s handling oficit.
If we are checking a term against aΠ-type that expects an implicit argument, but the term
is not a lambda, the system knows to automatically insert a metavariable (a "hole") to be
solved later.
•Clearer Error Messages:Because the system often knows what type it is "looking for,"
it can provide specific feedback when a term fails to meet expectations, rather than simply
stating that two complex inferred types do not match.
