Anonymity
The University Regulations makes it clear that all assessments are marked anonymously (Regulation
3.4) unless the assessment itself renders anonymity impossible e.g. placements, presentations,
practical assessments.  Unless otherwise stated, do not put your name or Banner ID on your
submission.
Referencing
The standard referencing style at UWS is Cite Them Right (CTR) Harvard and your references should
be in accordance with these guidelines.  The University provides a drop-down menu that lets you
see examples of how you should reference journals, books, websites etc.  Guidance on
referencing styles can be found on the UWS Library website:  https://uws-
uk.libguides.com/referencing
Extenuating Circumstances (ECS)
The University recognises that, from time to time, you may encounter issues which may prevent
you from being able to submit or undertake an assessment. Where this is the case, you can
complete an Extenuating Circumstances Submission (ECS) for consideration.  The ECS will be
forwarded to the School Assessment Board to take account of this declaration in recording your
module marks.  Guidance on ECS claims can be found: https://www.uws.ac.uk/current-
students/supporting-your-studies/exams-assessment-appeals/academic-appeals-extenuating-
circumstances/

Assessment guidance
Course Work 01 (CW01) – 100% of the total Mark
Note: Provisional and subject to change. If any changes occur, you will be notified accordingly.
Submission Overview:
This assignment consists of two parts: A Formal Documentation part submitted in written format
(Part A) and a Practical Work part (Part B) assessed via a demonstration.
The assignment parts are as follows:
• Part A - 40%: A written report documenting your work.
• Part B - 60%: A technical demonstration of your practical work assessed during a scheduled
presentation.
This assessment makes up 100% of the module mark. Per the assessment regulations, the pass mark
for undergraduate students is 40%.
Submission Instructions: Part A - Written Component - Formal Documentation (40%)
The written component must be submitted via the Aula (Turnitin) assignment box titled “Part A:
Written Component” and must include:
• Your Student Information: Include vital details such as your name, Banner ID, and any
necessary usernames/passwords that an examiner might need to access or evaluate your
work.
• Link to Your GitHub Repository: Provide the link to your project's GitHub repository containing
the source code. Please note that commits made to the repository after the submission
deadline will not be considered for assessment. Ensure your README.md file has
comprehensive instructions to clone, install, and run your application.
• Content: You should document the development process and include the following: a clearly
defined topic or feature covered during this module; an introduction to your project explaining
how the chosen technical feature contributes to its successful completion and performance; a
breakdown of your system architecture and functionality, supported by reproducible
examples to enhance understanding; a step-by-step tutorial guiding the reader on how to use
your project; and a conclusion with overall recommendations, summarising lessons learned
and providing actionable insights.
Format requirements: Part A - Written Component - Formal Documentation (40%)
1. Your submission can be either in a word .docx file or a PDF.
2. Your submission file name must follow this format: DWT-CW01-T2-Attempt-B123456-Your-
Name.docx
i. *Replace B123456 with your matriculation number (banner ID) and include your full
name instead of "Your-Name".
The report should be between 10 and 15 pages in length. If you require additional pages, please
consult your lecturer. (+- 10% is allowed without previous consultation.)

Note: Provisional and subject to change. If any changes occur, you will be notified accordingly.

Assessment Breakdown: Part A - Written Component - Formal Documentation (40%)
This assessment evaluates your ability to clearly document, explain, and critically reflect on the
development of a React-based mobile application (Progressive Web App or React Native). The
written submission should demonstrate not only what you built, but why and how it was designed,
implemented, and used.
Your report will be assessed using the criteria below, which directly correspond to the marking
rubric.
1. Presentation & Visual Appeal (Max. 5 Marks)
Your report should be professionally presented and easy to read. This includes:
• Clear structure with appropriate headings and subheadings.
• Consistent formatting, layout, and visual hierarchy.
• Effective use of diagrams, screenshots, or figures where appropriate to support
explanations.
2. Introduction & Feature Explanation (Max. 10 Marks)
You must clearly introduce your project and the key technical feature or topic(s) you have
chosen from the module content.
This section should include:
• A concise introduction to the application and its purpose.
• Identification of one or more specific React or mobile development features covered in the
module (e.g. React Hooks, PWA features, CRUD operations, storage solutions, hardware
APIs, Expo, APK generation, or a pre-approved alternative).
CEPS  5 Assessment Guidance Coversheet
• An explanation of how the chosen feature(s) contribute to the functionality, performance,
or user experience of the application.
• Practical examples illustrating how and where the feature is used within your project.
3. Technical Understanding & Documentation (Max. 10 Marks)
This section assesses your depth of technical understanding and ability to explain implementation
decisions.
You should:
• Describe the system architecture and main components of your application.
• Explain how the selected feature(s) have been implemented, supported by relevant code
snippets or pseudo-code where appropriate.
• Demonstrate understanding of how different parts of the application interact.
• Use clear, reproducible examples to support your explanations.
The emphasis is on explaining design and implementation choices, not simply listing technologies.
4. Project Setup & User Guide (Max. 10 Marks)
You must provide a clear, step-by-step guide that enables another user or developer to install
and use your application.
This should include:
• Instructions on how to download, install, and run the application.
• A link to your public code repository (e.g., GitHub), (or Expo Snack link, if applicable).
• Clear guidance on any required tools, dependencies, or configurations.
• A user guide explaining how to use the main features of the application.
Additionally, your Git repository must include a README.md file containing:
• Installation instructions.
• Usage guidelines.
• Any additional setup notes required to run the project successfully.
5. Critical Thinking, Conclusion & Recommendations (Max. 5 Marks)
The final section should demonstrate reflection and critical thinking.
You should:
• Summarise the overall outcome of the project.
• Reflect on challenges encountered and how they were addressed.
• Identify lessons learned during development.
• Provide recommendations for future improvements or extensions to the application.
This section should show evidence of evaluative thinking, not just description.
