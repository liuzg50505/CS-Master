Version Control Systems
Collaboration and development tracking tools!
xkcd comic
Version Control System (VCS)
● Track changes and maintain the history of changes
● Facilitate collaboration
Working by yourself?
● Look at old changes you made on the project
● Keep a log of why certain changes were made
● Work on parallel branches of development
Working in a team?
● Seeing what others have changes
● Resolving conflict in concurrent development
● Who wrote this ? When was this line edited ? by whom? Why?
● When, or after which change, the software broke? Why?
Types of version control systems
• Local data model: Maintains track of files within the local system
• Client-server model: Maintains track of file under the centralised server
(what if the server dies?)
• Distributed model: Any of the client can have a copy, rather than a single
central repository
Git
A distributed model which allows each developer works
directly on their own local computer, and changes are
shared between local computers as a separate step.
Github
A code hosting platform for Git version control and
collaboration.
Popular Jargons
● Repository: Entire collection of files and folders associated with a project
(remote -> server ; local -> your computer)
● Commit: Your submitted changes
● Branch: Repository can be organised into multiple lines (similar to railroad
track)
● Pull: Get the latest changes from the server (remote repository) and update
your local machine code (local repository)
● Push: Uploading your local machine code (local repository) to the server
(remove repository)
● Clone: Get a copy of a project from a remote repository on your computer
● Fork: Create a copy of someone else’s project into your own repository

Scenario – local branches
● Create github remote repository, clone to machine
● MAIN BRANCH
● Create/modify files, add+commit to local repository
● Push local changes to remote repository
● FEATURE BRANCH
● Create a new feature branch, switch to new branch
● Create file, add+commit to feature branch in local repository
● Merge feature branch into main branch in local repository
● Delete feature branch
● Push local changes to remote repository

Scenario – remote branches
1. Kevin creates a repository on Github
2. Kevin creates a project and links it to the repository which he has created
3. Kevin adds Andrew as collaborator to add some features to the software
4. Andrew clones the project
5. Kevin makes some changes to the source code
6. Kevin pushes his changes to the remote repository
7. Andrew updates his local repository by pulling the remote
8. Andrews creates a new branch (name it feature)
9. Andrew makes changes and commit/pushes them to the remote repository
10.Andrew merges his branch to the main branch
11.Kevin updates his branch to include the feature Andrew has added to the
project
12.Misc: .gitignore, squash, rebase etc.
Link Approach: Create a repository on Github
Create a project and link it to the remote repository
1. Initialise a git file
1. > git init
2. Add your files to staging area (use git status to see untracked and tracked
files)
1. > git add <files>
3. Commit your changes
1. > git commit -m “commit message”
4. Link the local repository to the remote repository
1. > git remote add origin URL
5. Push your changes to the main branch
1. > git push -u origin name-of-main-branch
Add collaborator and clone the project
Kevin makes changes and upload them
Andrew updates his local repository
git pull
Andrew create new branch and names it feature
1. Create a new branch
1. > git checkout -b “name-of-branch”
2. Switch to another branch
1. > git checkout name-of-branch
3. Check what branch you are on
1. > git branch
4. To push changes for the first time
1. > git push --set-upstream origin feature
Andrew merges feature branch to the main branch
1. Switch to the main branch
1. > git checkout “name-of-main-branch”
2. Merge branch to the main branch
1. > git merge name-of-branch
3. Push the changes
1. > git push
