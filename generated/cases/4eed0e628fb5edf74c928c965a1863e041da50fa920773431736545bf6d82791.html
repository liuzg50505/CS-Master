Our AI writing assessment is designed to help educators identify text that might be prepared by a generative AI tool. Our AI writing assessment may not always be accurate (it may misidentify
writing that is likely AI generated as AI generated and AI paraphrased or likely AI generated and AI paraphrased writing as only AI generated) so it should not be used as the sole basis for

Frequently Asked Questions
How should I interpret Turnitin's AI writing percentage and false positives?
The percentage shown in the AI writing report is the amount of qualifying text within the submission that Turnitin’s AI writing
detection model determines was either likely AI-generated text from a large-language model or likely AI-generated text that was
likely revised using an AI paraphrase tool or word spinner.
False positives (incorrectly flagging human-written text as AI-generated) are a possibility in AI models.
AI detection scores under 20%, which we do not surface in new reports, have a higher likelihood of false positives. To reduce the
likelihood of misinterpretation, no score or highlights are attributed and are indicated with an asterisk in the report (*%).
The AI writing percentage should not be the sole basis to determine whether misconduct has occurred. The reviewer/instructor
should use the percentage as a means to start a formative conversation with their student and/or use it to examine the submitted
assignment in accordance with their school's policies.
What does 'qualifying text' mean?
Our model only processes qualifying text in the form of long-form writing. Long-form writing means individual sentences contained in paragraphs that make up a
longer piece of written work, such as an essay, a dissertation, or an article, etc. Qualifying text that has been determined to be likely AI-generated will be
highlighted in cyan in the submission, and likely AI-generated and then likely AI-paraphrased will be highlighted purple.
Non-qualifying text, such as bullet points, annotated bibliographies, etc., will not be processed and can create disparity between the submission highlights and the
percentage shown.

Introduction
The campus events should have broader participation. The User entity includes the user’
s name, email, encrypted password, and role (admin or regular user). The Organizations
entity represents the many campus groups or departments that plan events, including
information such as name, description, and contact information. The Activity entity is the
heart of the system and stores information such as title, description, date, location, total
number of participants allowed to sign up, and a Foreign Key linking to the Organization
that is organizing the activity.
Entity Relation
Data Management is accomplished through the design of an Entity Relationship (ER)
where there are a total of six entities within the database, which allows for the creation of
relationships between Entities, as displayed through a comprehensive ER Diagram
shown in (Figure 1).

Contact Information. Activities are the core of the Database and contain the following

Technologies Used
PHP
The primary server-side scripting language for this application is PHP. PHP is compatible
with virtually all Web Servers and is easily integrated with MySQL Databases. The User
Friendliness and Extensive Documentation greatly contribute to the usability of PHP as a
solution for core functions of the application for functions such as User Authentication,
Form Processing, and Database interaction. All credentials are stored using the
Password Hashing Functions Present in PHP in line with best practices for Data
Protection.
MySQL
MySQL was selected as the Database Management System (DBMS) given its proven
Reliability and Performance features, and, once again, its full compatibility with PHP
applications. MySQL is an Open Source Relational Database Management System that
supports Structured Query Language (SQL) very well. The Application uses MySQL as its
sole source for all Entity Data Storage. To assure Referential Integrity, Foreign Key
Constraints will be used to maintain Referent Integrity across tables that will
cross-reference one another. the Database Design will support simple queries related to
User interaction features, and more complex joins related to Administrative Reporting.

HTML&CSS
The Frontend uses Pure HTML with Inline CSS Styling to provide a Simple and
Lightweight User Interface, with complete compatibility across Basic Web Server
Configurations, Fast Load Times, and Broad Access on Different Browsers/Devices. The
System is designed so that Semantic HTML Structure provides Clarity and Maintains
Accessibility Through Minimal Use of Inline Styling, Creating a Cleanly Functional
Interface Both for Users and Administrators.In this case, the Apache web server was
selected because it had been in use for many years and had widespread use of mod_php
support for the processing of PHP scripts. Because of Apache's capabilities to process
multiple simultaneous user requests, it could handle a significant amount of load from
users accessing the service at the same time.
Development Process
During the initial stages of development, the database was designed to meet the needs
of the PHP application; in particular, the six core domain objects that constitute the
application were identified, and the relationships between them were mapped out. Once
that step was completed, a MySQL Database Schema was created that was designed to
ensure the integrity of the object relationship with proper Foreign Key Relationships. Test
data was also inserted into the database to test the application's core functionality.
Once the Database was ready for use, the key components of the application were
developed, beginning with the Database Connection and the user Authentication
Processes. The Registration and Login modules were developed with front-end
validation to guarantee the completeness and accuracy of the submitted information. As
shown in Figure 2 below, the registration interface is designed to provide a visual
