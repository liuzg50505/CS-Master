Lecture 4: LLMs for Software Engineering
Dr Paul Henderson
University of Glasgow

What is an LLM?
● "LLM" = large language model
○ 'language' ⇒ trained on text
○ includes ChatGPT , Gemini, Doubao…
○ …but also code-speciﬁc models (Codex, Claude Code, …)
● today's goal = learn to use LLMs more effectively for coding
What is an LLM?
● LLMs predict the likely next 'token' of some text/code
○ token = "word" or "symbol" (roughly)
○ input many past tokens, predict next one
○ …and repeat
○ learnt from a big dataset
What is an LLM?
● LLMs predict the likely next 'token' of some text/code
○ token = "word" or "symbol" (roughly)
○ input many past tokens, predict next one
○ …and repeat
○ learnt from a big dataset
● they "guess" what's most likely to come next
● they often guess right
● they sometimes guess wrong
○ e.g. obscure topics, unusual phrasing
What is an LLM?
● outputs are partially random
○ predict N plausible options for the next token
○ choose one randomly
What is an LLM?
● outputs are partially random
○ predict N plausible options for the next token
○ choose one randomly
● hallucination = mistake
○ plausible (usually) but wrong
○ not a "misunderstanding"
○ might hallucinate a non-existing library function
○ might hallucinate an error in code where there's none
● LLMs are (usually) over-conﬁdent
LLMs are great for…
● explaining unfamiliar code
● generating boilerplate
○ i.e. simple, standard code
● implementing 'small' functions
○ 10s–100s of lines; not too 'esoteric'
● translating between languages
● suggesting test cases
● refactoring
When not to use them…
● security-critical and safety-critical code
● API usage
○ e.g. paid or untrusted services
● critical business logic
● legal or licensing decisions
● expert code-review by a human partially mitigates 1st three
Prompting
● be explicit
○ include relevant context
"write a function to read data"
"write a python function that loads a csv ﬁle from a
user-speciﬁed ﬁlename, returning a pandas dataframe"
Prompting
● "what could go wrong?"
● "are there any edge cases?"
● questions about design choices
○ "why do you use python lists instead of numpy arrays?"
● questions about behaviour in speciﬁc cases
○ "will this be slow for large arrays?"
○ "will it crash if the user gives invalid input?"
Trust, but verify
● verify everything important!
○ same as if you asked some random guy to write your code
● run the code
○ including edge-cases
● if the LLM wrote you a test, then test the test
○ there's no more reason to assume the test is correct than the original code
Trust, but verify
● think of the LLM as a junior assistant
○ fast, energetic, sometimes wrong
● human (you!) = responsible engineer
○ domain knowledge, judgement, accountability
● expert human + LLM = power
○ non-expert + LLM = replaceable
Context management
● 'context' = all the previous stuff in the conversation
○ everything the LLM 'sees'
○ limited resource! typically ~200K tokens
● theory: more = better, LLM can think about more code
Context management
● 'context' = all the previous stuff in the conversation
○ everything the LLM 'sees'
○ limited resource! typically ~200K tokens
● theory: more = better, LLM can think about more code
● practice: LLMs degrade beyond ~50K tokens
○ start to get 'confused' by their own past thoughts
○ cannot isolate details in the chat
○ self-reinforce instead of ﬁxing errors
Context management
● keep tasks in a session short and contained
● plan / brainstorm in a separate chat
○ providing all maybe-relevant existing code
● ask clarifying questions in a separate chat
○ e.g. suitable approaches / libraries for sub-tasks
● …then feed the results back into a new chat
Chat vs agents
● chatbot = you type something, AI replies
○ …then you decide what to do next – test the result / ask more / …
● agent = AI does stuff autonomously
Chat vs agents
● chatbot = you type something, AI replies
○ …then you decide what to do next – test the result / ask more / …
● agent = AI does stuff autonomously
○ it decides what to do next
○ it can run tools
○ it observes outcomes and behaves accordingly
● no magic – still just an LLM
○ called in a loop
○ can 'ask' for access to tools (e.g. edit/run code; search; etc.)

Agents
● user: "Add a button to show the list of all products"
● agent might…
○ read the code repo
○ edit several code ﬁles
○ create / modify tests
○ run tests, observe failures
○ ﬁx (re-edit) the code
○ repeat until tests pass
Agents
● advantages:
○ fast iteration
○ minimal supervision
● particularly useful for…
○ refactoring
○ test generation
○ adding clearly-speciﬁed features
● feels like a junior dev who never sleeps!
Agents
● risky:
○ makes many changes quickly
● can…
○ break working code
○ delete ﬁles
○ introduce bugs
